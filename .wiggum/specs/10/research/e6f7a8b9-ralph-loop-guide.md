# Research: Ralph Loop Guide

## Question

What does the ralph loop guide describe? What are its two modes, iteration patterns, prompt templates, quality checks, and configuration approach?

## Findings

### Two Operating Modes

- `./loop.sh [plan|build] [max_iterations]`
- **Plan mode**: feeds `PROMPT_plan.md`, studies gap between specs and code, produces `IMPLEMENTATION_PLAN.md` -- no implementation, no commits. Typically 1-5 iterations.
- **Build mode**: feeds `PROMPT_build.md`, picks one task, implements, validates, commits, exits. Typically 15-25 iterations.

### Build Iteration Pattern (10-Step)

1. Orient (subagents study specs)
2. Read plan
3. Select most important task
4. Investigate existing code
5. Implement
6. Validate (exactly 1 subagent for build/tests)
7. Update plan
8. Update AGENTS.md if needed
9. Commit + push
10. Session ends, context cleared

Key invariant: one-task-per-iteration. Quality degrades past 40% context utilization (~176K usable tokens). Fresh context per iteration keeps reasoning in the "smart zone".

### Prompt Templates

- Numbered step structure: `0a-0d` for orientation, `1-4` for main instructions, `99+` for guardrails, `999+` for hard constraints, `9999+` for critical invariants
- Load-bearing phrases: "study" (not "read"), "using parallel subagents", "only 1 subagent for build/tests", "ultrathink", "don't assume not implemented", "capture the why"

### Quality Checks (Backpressure)

- Shell commands returning exit 0 on success
- All acceptance criteria must map to a verifiable shell command BEFORE the loop starts
- Primary: deterministic -- `pytest`, `pyright`, `ruff check`, `curl`, `test -f`
- Secondary: LLM-as-judge with binary PASS/FAIL prompt
- Validation must use exactly 1 subagent to avoid conflicting signals

### Configuration (Five Files)

| File | Role |
|------|------|
| `AGENTS.md` | Build/test/lint commands, conventions, gotchas; max 60 lines; self-updating |
| `PROMPT_plan.md` | Plan mode prompt template |
| `PROMPT_build.md` | Build mode prompt template |
| `specs/*.md` | Durable requirements, one file per topic |
| `IMPLEMENTATION_PLAN.md` | Dynamic task tracker; disposable; generated by agent |

All file-based, git-tracked. No runtime configuration. All tuning happens by editing files. `AGENTS.md` is self-improving: build prompt instructs agent to update it with operational learnings.

### Advanced Patterns

- Work scoping at plan-creation time on a branch, not runtime (runtime task filtering only 70-80% reliable)
- Parallel instances via separate git worktrees
- Cron pattern: one iteration per night for incremental improvement
- Self-improvement loop (Boris Cherny): every observed failure becomes a rule in AGENTS.md or specs

## Key Design Insight

Bad specifications cannot be fixed by more iterations. The plan is the bottleneck, not the build loop.
